The approved player-facing release is a layman-readable `::duel` result that shows `COMBAT REPLAY VERIFIED` for eligible classic-formula No Magic stake duels. Its canonical context is v3 and its witness wrapper is v2. The private journal continues to show only the requester’s swings and combat modes; it never models or renders the opponent’s swing/style rows. The participant-authorized technical witness necessarily discloses both sides’ formula inputs, dynamic styles/prayer tiers, and seeds transiently to the local verifier, which is an explicit technical disclosure rather than part of the ordinary UI. A modified client can inspect those bytes despite buffer wiping.

Proof contributions use a true server + two-client commit/reveal/lock ceremony. The server durably commits its context-bound seed before clients are challenged. It then sends exact context-v3 chunks before client entropy; an honest client verifies and attests the locally visible subset of its own starting state, both visible combat levels, the displayed opponent identity, and both displayed stake offers before contributing a seed. Both client commitments are durable before either seed reveal; both reveals and a seed-free final lock are durable before both clients ACK; only then may combat start. PC/Android use `SecureRandom` and TeaVM uses browser crypto with no fallback. The current client cohort is `10136`. Hidden Quest controls retain the `@vsduelproof@v1` transport envelope and replies reuse `INTERFACE_OPTIONS` sub-option `19`; journal proof detail uses `@vsduel@v2`. No opcode was added. Magic-allowed and OSRS-melee-formula duels are ineligible, never partially green.

Slices 1–4 are complete. The shared dependency-free crypto/RNG module and durable server + two-client handshake guard entry through `LOCKED -> COMBAT`; eligible initiative always consumes committed draw one, gives the lower displayed/base combat level first swing, and uses that draw only for an exact level tie. Every later classic-melee random choice comes from the same private stream without shifted boundaries. Committed combat levels, gear, and formula-relevant Attack/Strength/Defence state are frozen during combat, while combat styles and permitted prayer toggles/tiers remain dynamic and are recorded per swing. Before lethal direct-melee or recoil damage, the server freezes witness v2 and independently replays every draw plus both players’ full HP path and capacity-bounded recoil from context v3, including direct-kill recoil suppression. It binds the exact draw/swing boundary, winner/cause/time, participant identities, both accepted stake lists, and every receipt swing. Poison, generic external damage, desert heat, cannon mutation, and stat restoration are paused or blocked during active proof combat. Fresh recoil capacity/shatter matches the verifier, and a worn ring cannot also be the physical staked ring unless No Weapons safely unequips it. After payout, receipt+witness+`COMBAT -> VERIFIED` commit atomically; database failure cannot create green. Client `10136` independently verifies the bounded witness, binds both stake lists, displayed identities/levels, and its private receipt rows, wipes the raw transfer, and retains only `VERIFIED` / `FAILED` / `UNAVAILABLE` plus proof id for the UI. It separately retains up to 16 seed-free lock anchors for the login session; bare latest `::duel` must match the newest anchor exactly, while reset/reconnect wipes them. Adjacent approach overlaps the handshake and the lock/combat markers commit in one transaction after both ACKs; a fallback preserves the old guarded approach. A legal allowed retreat is `ABORTED / RETREAT`, moves no stakes, creates no receipt, and retires only that terminal-less anchor; No Retreat continues to block walking. Focused shared/client/voidbot tests, full build, Android debug, TeaVM packaging, all three workbench proof cards, and live unequal/equal-level verified two-voidbot stake duels pass.

Honest claim boundary: the proof establishes committed RNG and HP/recoil replay conditional on the static context and the server-recorded dynamic style/prayer tape. It cannot cryptographically prove that every dynamic toggle was the exact live user action; the honest requester can compare its own private rows, but opponent action truth remains server-recorded. Without an external/public append-only transparency log it also cannot prove absence of selective abort, omission, or deletion, and it cannot make a maliciously modified client attest or report honestly. No website verifier, public log, or backfill is approved; the game has not opened.

Acceptance gotchas: checking `No Magic` on the duel rule screen is mandatory for a green proof; staking an item is optional. A Magic-allowed receipt correctly shows gray `COMBAT REPLAY UNAVAILABLE` with `No Magic: melee replay required`. `::duel` is blocked for the whole active-duel lifecycle with `blocked|duel_active` so an unfinished duel cannot open an older receipt and trip the newest-lock check. On PC, the Journal receives mouse presses synchronously so post-death input cleanup cannot swallow its visible X; Escape uses the same close path.

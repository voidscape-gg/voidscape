Strategic decision: the desired near-term end-state is Android-app parity on iPhone Safari, not an OSRS-style redesign. Keep shared renderer/protocol/game state; use the existing Android app as the source of truth for mobile input, panel access, chat, keyboard behavior, world map/walker, orientation, and touch semantics. Do not recreate the game renderer/assets in JS again.

HUD direction clarification from physical-phone planning: mimic the Android app first, then rearrange after real phone tests. Keep one owner per piece of HUD state. Java/shared client owns inventory, panels, chat history/tabs, minimap/world map, account/settings, and gameplay state. The browser shell owns only phone affordances such as safe-area layout, soft-keyboard focus, browser Back, diagnostics, and Safari-specific keyboard/context helpers. DOM launchers must not duplicate visible canvas tabs in normal play.

Current chat baseline: normal iPhone HUD uses `ui.chatAccessMode: "canvas"`. The real canvas chat/tabs own chat display and tab switching. Phone portrait keeps the framed Voidscape chat history, while phone landscape now uses `ui.phoneLandscapeLooseChat: true`: the large chat history frame/background is suppressed and recent chat draws as loose OpenRSC-style text above the shared bottom chat tabs. The old DOM chat helper/tray is hidden in normal Android-parity mode; the browser still owns the `Aa` soft-keyboard focus affordance and context/secondary helper because Safari needs a reliable bridge into shared input state.

Current panel baseline: the OSRS-style bottom canvas dock experiment is superseded. `mudclient` disables the TeaVM-only mobile panel shell for iPhone/Android-profile web mode and keeps the shared canvas top tabs as the active launcher, matching the Android app's "same game on another platform" behavior. `WebClientPort` publishes `ui.panelAccessMode: "canvas"`, `ui.mobilePanelShell: false`, `ui.canvasTopTabsVisible: true`, `ui.canvasPanelRailVisible: false`, and `ui.canvasPanelDockVisible: false`. The DOM `UI` button/drawer stays hidden in normal play as fallback/debug wiring.

Current resize/orientation baseline: TeaVM phone portrait keeps the Android-like `512` logical width and expands full height by viewport aspect up to the existing `1152` cap. Phone landscape keeps the classic `346` full height but now computes logical width from the viewport aspect (`round(346 * cssWidth / cssHeight)`, clamped at `1152`) so Mobile Safari fills the available width instead of leaving side gutters. This is still the shared Java renderer and HUD, not a separate JS renderer. Desktop and tablet runtime modes stay deliberate branches and do not show phone-only controls by default.

Current full-surface touch baseline: TeaVM listens on the full game surface around the canvas, not only the visible canvas element, while still converting through the canvas rect.
